# 1.16 — Observed-candidate relay-only enforcement (F20)

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F20; scope-corrected from F11 wire + F20) ·
**Depends on:** 1.15, 1.14 · **Status:** done

## Goal
The relay-only privacy guarantee is enforced from **observed** gathered candidates rather than derived
only from policy — a real transport backend now exists (1.15), so F20's "when a real backend lands"
condition is met.

## Scope
- **In:** classify actual gathered ICE candidates and abort the dial if a host/srflx candidate appears
  under relay-only, replacing the policy-derived privacy line in `apps/core/src/session.rs` with
  observation-based enforcement.
- **Out (re-scoped — see Status):** the netns/tcpdump wire-level acceptance matrix against a real
  backend + coturn — this task's original deliverable 1 — turned out to need its own CLI transport-
  selection wiring plus a multi-namespace/coturn/tcpdump harness comparable in size to T04/T05's
  original estimate, not an extension of F20's self-contained fix. Split into
  [1.22](./1.22-webrtc-cli-transport.md) (CLI `--transport webrtc` wiring) and
  [1.23](./1.23-netns-nat-matrix.md) (the netns/tcpdump matrix itself, depends on 1.22). Media (Feature
  10) stays out of both.

## Deliverables
1. Candidate-classification code that aborts a relay-only dial on any non-relay candidate + its test.
2. Updated `apps/core/src/session.rs` relay-only UX line (`session info`'s `candidates offered:` claim)
   derived from observed candidates, not `policy` alone.

## Risks / notes
This is the invariant that makes "relay-only" a real IP-hiding guarantee, not a label — the abort is
**fail-closed** (any host/srflx candidate under relay-only aborts, never downgrades silently, and an
unrecognized/unparseable candidate line is treated as worst-case rather than passed through).

**Doc correction:** the task file and review-report.md F20 both cite
`apps/rendezvous/src/session.rs:296-299` as the line to fix. That path does not exist —
`apps/rendezvous/src/` has no `session.rs`, and the crate never touches ICE candidates or relay policy
(it only mints TURN credentials). The actual `candidates offered:` line has always lived in
`apps/core/src/session.rs` (the P2P session substrate), which is what this task fixes. Read as a stale
path from when the finding was drafted, not a real gap in rendezvous.

## Tests
`cargo test -p meridian-core` (unit tests for `relay::candidate_class`/`observed_classes` and
`session::enforce_relay_only`; an end-to-end `dial_with_config`/`answer_with_config` integration test
over `LoopbackTransport` proving a real relay-only session's `session info` reflects observed, not
assumed, candidate classes; and two genuine end-to-end abort tests — `LeakyTransport`, a
`LoopbackTransport` wrapper that deliberately reports a leaked host candidate regardless of policy,
drives both `dial_established` and `answer_established` through the real `RelayOnlyViolation` abort,
with a `CountingRelay` wrapper proving the abort happens before the offer/answer ever reaches
signaling, not merely that the call eventually returns `Err`) · `cargo clippy -p meridian-core -p
meridian-transport --features webrtc --all-targets -- -D warnings`.

## Reviews
**security-reviewer** (relay-only IP-hiding invariant, fail-closed), **connectivity-debugger** (candidate
classification correctness against both the loopback and real candidate-string formats),
**test-engineer**.

## Status (done)
`apps/core/src/relay.rs` gained `CandidateClass`, `candidate_class()` (parses both the standard ICE-SDP
`typ host/srflx/relay` form produced by the real webrtc-rs backend and `LoopbackTransport`'s
`candidate:host ...` shorthand with one classifier), and `observed_classes()` (folds a candidate list
into a `GatherClasses`, treating anything unparseable as worst-case `host+srflx`).
`apps/core/src/session.rs` gained `SessionError::RelayOnlyViolation` and `enforce_relay_only(policy,
&candidates)`, called in both `dial_established` and `answer_established` immediately after gathering
local candidates and **before** the offer/answer is sealed and sent to the peer — so a violation aborts
before any address could leak, not after. `P2pSession` now carries `offered: GatherClasses` computed
once from the real gathered candidates at connection time; `SessionInfo::offered` (the `session
info`/`doctor` "candidates offered" line) reads that instead of recomputing `relay::gather_classes
(policy)` after the fact — confirmed by tracing `apps/cli/src/doctor.rs`'s relay-only line, which
establishes a real session and reads `info.candidates_offered_line()`, i.e. observed data.
`relay::gather_classes` itself is kept for a genuine future before-a-session-exists use (a settings UI
previewing what a policy *would* gather); its doc comment was corrected after review to stop claiming
`doctor` uses it that way — today its only real caller is this module's own tests.

All new tests pass (10 new unit tests across `relay.rs`/`session.rs`, three new end-to-end integration
tests — including two added after test-engineer review that drive the abort through a real
`dial_established`/`answer_established` call via a `LeakyTransport` double, with a `CountingRelay`
proving the ordering guarantee), full workspace build is clean, `cargo clippy -p meridian-core -p
meridian-transport --features webrtc --all-targets -- -D warnings` is clean, `cargo fmt` is clean, and
the full gated `--features webrtc` suite (including 1.15's real ICE/SCTP/DTLS tests) still passes with
no regressions.

Reviewed by security-reviewer, connectivity-debugger, and test-engineer — all APPROVE, no blocking
findings. security-reviewer flagged one medium, non-blocking residual gap worth tracking: the
classifier trusts a candidate's self-declared `typ` label as ground truth with no independent
cross-check (e.g. against the configured TURN server host, or private/loopback address ranges) — a
bug that made webrtc-rs's ICE agent gather a real host/srflx candidate but mislabel it `typ relay`
would pass silently. This doesn't undermine the primary construction-level guarantee
(`RTCIceTransportPolicy::Relay`, which prevents such candidates from being gathered in the first
place) and is out of this task's narrower stated scope (parse-and-abort on the transport's own
declared type), but is recorded here rather than silently dropped — a natural follow-up alongside
1.23's real packet captures, which would catch this class of bug at the wire level regardless of what
the candidate string claims. connectivity-debugger separately flagged a low-severity, fail-safe-only
nit (a TURN hostname literally equal to `typ` could make `LoopbackTransport`'s shorthand-format
fallback misclassify as unparseable, causing an unnecessary abort, never a leak) — not fixed, judged
not worth the complexity for an unrealistic hostname choice. Also fixed after review: two stale
doc-only claims (`relay::gather_classes`'s comment, and `docs/security/threat-mitigation-matrix.md`'s
A4/A5 "Proven by" column, which cited only the still-pending T05/1.23 packet-capture proof without
crediting this task's now-landed code-level enforcement).
