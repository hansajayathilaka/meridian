# 1.26 — Drive real `meridian` peers across the topology + capture pcaps

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire, prerequisite) · **Depends on:** 1.24, 1.25 · **Status:** done

## Goal
Two real `meridian` processes, one in `ns-alice` and one in `ns-bob`, actually establish a session across
each of the four NAT cells via `ip netns exec`, using [1.24](./1.24-real-signaling-p2p-cli.md)'s
`session connect` command and [1.25](./1.25-netns-topology-coturn.md)'s topology, with `tcpdump`
capturing at `ns-bob` and `ns-turn` for every run.

## Scope
- **In:** wire `tools/netns-nat-matrix.sh`'s `cell()` to launch both peers (`ip netns exec ns-alice …
  session connect …` / `ip netns exec ns-bob …`) against 1.25's in-topology rendezvous, selecting the
  relay policy per cell (direct/prefer-relay/relay-only per the matrix's intent); start/stop `tcpdump` on
  `ns-bob` and `ns-turn` bracketing each run; save one pcap per cell per capture point with a clear,
  predictable naming scheme for [1.27](./1.27-pcap-assertions-ci.md) to consume; print the negotiated
  path (`session info`) per cell so a human can eyeball correctness before 1.27 automates it.
- **Out:** the strict pass/fail pcap assertions themselves and CI wiring (→
  [1.27](./1.27-pcap-assertions-ci.md)) — this task proves peers connect and pcaps exist, not yet that
  the pcaps prove the privacy/relay claims.

## Deliverables
1. `full-cone`, `port-restricted`, and `symmetric:symmetric` (3 of 4 cells) each produce a successful
   `session connect` across the real topology. The 4th cell (`udp-blocked` + `relay-only`) fails fast
   (~20–50s, well inside the harness's 90s outer bound) with the documented, architect-confirmed
   `webrtc-ice` TURN-over-TCP dependency ceiling (1.30) rather than connecting — this is an explicit,
   recorded amendment to the original criterion (architect sign-off, 2026-07-23), not a silent downgrade
   of scope.
2. A pcap file per (cell × capture point) saved to a predictable path/naming scheme.

## Risks / notes
This is where 1.24's cross-process assumptions get exercised for real (real ICE gathering behind real
NAT, not localhost loopback-address ICE) — expect to find and fix small connectivity issues here; route
those through **connectivity-debugger**, not silent workarounds. If timeouts differ meaningfully from
localhost, note it but don't relax `dial_with_config`/`answer_with_config`'s existing bounded-wait
behavior to "fix" it — a real timeout under `udp-blocked` is a legitimate slow path, not a bug to paper
over.

## Tests
`sudo bash tools/netns-nat-matrix.sh cell <name>` for each of the four cells, asserting a successful
connect + non-empty pcaps (root/`NET_ADMIN` required).

## Reviews
**connectivity-debugger**, **test-engineer**.

## Status (done)
`tools/netns-nat-matrix.sh`'s `drive_real_peers()` (wired into `cell()` after `wire_smoke_cell`) does
everything this task's scope asked for and works correctly end-to-end: idempotent alice/bob identities
under the rig dir, per-cell relay-policy selection (`direct` for full-cone/port-restricted, `prefer-relay`
for symmetric, `relay-only` for udp-blocked), tcpdump bracketing at `ns-bob`/`b-eth` and `ns-turn`/`t-eth`
with predictable pcap naming (`$d/pcaps/<cell>-{bob,turn}.pcap`), `session info`-style path/transport/role
summaries parsed from each side's `--json` output, and a loud non-zero-exit failure path with full
stdout/stderr dumped when either side doesn't establish. Independently verified: the harness mechanics
(topology, coturn, rendezvous, identities, policy config, tcpdump start/stop, pcap production, teardown)
all work correctly regardless of whether the underlying P2P connection itself succeeds.

**Deliverable #1 ("all four cells produce a successful `session connect`") is currently unmet in all four
cells** — this is a real, hard blocker per the architect's review (not a "premise didn't hold" situation
like 1.23's split; the harness itself is proven, the P2P connections it drives are not). connectivity-debugger
root-caused two distinct real bugs against the live rig (pcap decoding, live re-testing with a temporarily
bumped-then-reverted `WAIT_TIMEOUT`, and direct reads of the pinned `webrtc-ice` 0.17.1 source):

- **Bug A** (`full-cone`/`port-restricted` under `direct`, `symmetric:symmetric` under `prefer-relay`):
  under `RTCIceTransportPolicy::All`, ICE never nominates a candidate pair (`transport error: no candidate
  pair selected yet`) even though `RelayOnly` succeeds cleanly in the identical topology — the permanently-
  unreachable host/srflx pairs never reach a terminal `Failed` state that would let the agent fall through
  to the (working) relay-vs-relay pair, per `webrtc-ice-0.17.1`'s `agent_selector.rs` nomination logic.
  Architect's recommendation: try tightening `SettingEngine::set_ice_timeouts` in
  `apps/transport/src/webrtc_backend.rs::new_session` first (config-level, no design-doc change needed)
  before reaching for a session-level policy-retry fallback (which would need a
  `seq-call-relay-fallback.mermaid` + Feature 5 spec update and architect sign-off, since the diagram
  documents a single-phase fallback, not a two-phase retry).
- **Bug B** (`udp-blocked` under `relay-only`): hangs completely (no output, killed only by the script's
  outer 90s timeout). Root cause confirmed by reading `webrtc-ice-0.17.1`'s
  `agent_gather.rs::gather_candidates_relay` directly: TURN-over-TCP client support does not exist in this
  pinned version at all (`transport=tcp`/`turns:` URLs hit an explicit `else` branch with a `TODO` comment
  acknowledging the gap) — no wiring change in `session_connect.rs`/`webrtc_backend.rs` can fix this.
  Architect's disposition: a semver-compatible `webrtc`/`webrtc-ice` bump (if one adds TURN/TCP) is a
  routine dependency update, no ADR needed (these crates aren't on stack.md's wire-critical pin list); if
  no upstream fix exists yet, document `udp-blocked + relay-only` as a known unsupported combination.

Both fixes were carved into their own tracked tasks — **[1.29](./1.29-ice-nomination-relay-fallback.md)**
(Bug A) and **[1.30](./1.30-turn-tcp-dependency-gap.md)** (Bug B) — rather than folded into this task's
scope, mirroring how 1.28 was carved out during 1.23's split. Both are now done.

**Re-run against both fixes, independently verified**: `full-cone`, `port-restricted`, and
`symmetric:symmetric` all report `ALL CHECKS PASSED` with `session connect` actually establishing
(`path=relay`, `relay_fallback:true`) against the real rig. `udp-blocked` still cannot connect — but now
fails fast (~20-50s) with a clear error instead of hanging, per 1.30's fix.

**Completion-criteria amendment (architect sign-off, 2026-07-23):** my original review held "all four
cells connect" as a hard, literal gate. Once 1.30 established — by direct source diff against the only
extant newer `webrtc-ice` release (0.17.2), not speculation — that client-side TURN-over-TCP support does
not exist at **any** available version of the pinned dependency, holding this task to that literal reading
would require an outcome that is provably unachievable at the current dependency pin, which is a different
(and unreasonable) thing to hold open indefinitely versus a genuine unresolved bug. Requiring the
impossible is not what the hard gate was meant to protect against — it existed to prevent quietly waving
away a bug that might still be fixable, and this is no longer that situation. Deliverable #1 above and this
Status are the explicit, recorded amendment (not a silent reinterpretation) that resolves this task as
**done**: 3/4 cells connect for real; the 4th is a documented, architect-confirmed upstream limitation, not
a task failure or an unresolved Meridian bug.

**Note for [1.27](./1.27-pcap-assertions-ci.md):** per the same architect review, 1.27's pcap assertions
should split by what each assertion presupposes rather than being all-four-strict or three-only-scoped —
assertions presupposing an established path/data flow (negotiated-rung matching, DTLS-ciphertext-only at
`ns-turn`) scope to the 3 connecting cells; the `relay-only` zero-host/srflx-leak assertion is the
anonymity-model "must never" invariant and should still run against `udp-blocked`'s pcap too (it must hold
trivially there, since 1.16's `enforce_relay_only` means host/srflx are never gathered regardless of
whether the session ultimately connects — verifying this holds even during a failed attempt is a stronger
proof, not one to skip); add one narrow assertion specific to `udp-blocked` proving the "fails fast and
loud" claim on the wire (no established data channel/candidate pair, matching the bounded 1.30 failure
mode), not just via CLI stdout.
