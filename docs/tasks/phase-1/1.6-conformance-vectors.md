# 1.6 — Conformance vectors: X3DH / ratchet / envelope / safety numbers + CI runner

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F1) · **Depends on:** 1.1 · **Status:** done

## Goal
Known-answer vectors freeze the crypto wire behavior so a spec-divergent KDF label fails CI instead of
surfacing as a silent interop break when Features 11/12 (browser/mobile) implement the same spec. This is
the single highest-leverage test gap and the stated precondition for the external crypto review.

## Scope
- **In:** generate via `xtask -- vectors` and commit to `test-vectors/`: X3DH intermediates
  (DH legs, concatenation, derived root key), a fixed Double Ratchet transcript (sends/receives incl.
  skipped keys + header encryption), envelope encode/decode vectors, and safety numbers (fill the empty
  `test-vectors/safety-numbers-v1.json`). Replace the `TODO` conformance step at
  `.github/workflows/ci.yml:51` with a real runner that checks the crate against the committed vectors.
- **Out:** browser/mobile consumers (Features 11/12); the external crypto-review engagement itself.

## Deliverables
1. `xtask -- vectors` subcommand generating the four vector sets.
2. Committed vector JSON under `test-vectors/` (X3DH, ratchet, envelope; filled safety-numbers-v1.json).
3. A conformance test that fails on a deliberately spec-divergent KDF label.
4. Wired CI conformance job (replacing the `ci.yml:51` TODO).

## Risks / notes
Vectors must be generated from the frozen v:1 spec, not just whatever the code emits — a self-consistent
vector that bakes in a bug is worthless. Cross-check labels/ordering against the report's verified
description of the construction. Coordinate with 1.5 (byte-identical after zeroization) and 1.1 (frozen
decision recorded in ADR 0015). Precondition for Verdict step 5's external review — do not let it slip
behind Group C.

## Tests
`cargo nextest run -p meridian-crypto` (vector tests) · the new CI conformance job locally.

## Reviews
**security-reviewer** (vectors faithful to spec), **test-engineer** (CI runner + failure mode).

## Result
Added byte-pinned conformance vectors for X3DH, the Double Ratchet, `MessageEnvelope`, and safety
numbers, generated by `cargo run -p xtask -- vectors` into `test-vectors/{x3dh,ratchet,envelope,
safety-numbers}-v1.json` (identity-v1.json unchanged), plus a new `apps/crypto/tests/conformance.rs`
that re-derives every vector via the crate's real public API and asserts byte equality — the actual
CI gate, not just "the generator ran." `xtask`'s single-file `vectors.rs` was split into
`tools/xtask/src/vectors/{mod,identity,x3dh,ratchet,envelope,safety_numbers}.rs`.

**Determinism boundaries** (documented, not worked around): header-encryption ciphertext bytes are
never pinned (`header_seal` draws a random 24-byte nonce by design) — only a functional round-trip
vector is recorded; a handful of ratchet chain-key values seeded by the *receiving* side's
internally-CSPRNG'd keypair (the actual PCS mechanism) are marked `chain_key_pinned: false` rather
than falsely pinned. Both are called out in the vector JSON `note` fields and in code comments so
they're never mistaken for coverage gaps.

**Visibility widening (flagged for and cleared by security-reviewer):** to drive vector generation
through the crate's *real* derivation code instead of a parallel reimplementation in `xtask` (which
could itself drift from spec), `dh`, `ed25519_pub_to_x25519`, `kdf_rk`, `kdf_ck`, `header_seal`,
`header_open` moved from `pub(crate)` to `pub` in `primitives.rs`, reachable from outside the crate
only via a new `#[doc(hidden)] pub mod test_support` re-export (`mod primitives` itself stays
private). `X3DH_INFO`/`RK_INFO`/`MSG_INFO`/`hkdf`/`aead_seal`/`aead_open`/`gen_dh` were deliberately
left `pub(crate)`. `ratchet.rs` gained `DoubleRatchet::init_initiator_with_keypair` (identical to
`init_initiator` but takes the sending secret explicitly instead of drawing it from the OS CSPRNG;
`init_initiator`'s real RNG behavior is unchanged, it now just delegates) — marked `#[doc(hidden)]`
per security-reviewer's nit so it doesn't surface in `cargo doc`/autocomplete for future browser/
mobile consumers.

**F1 proof:** `x3dh::tests::divergent_kdf_label_does_not_match_committed_vector` flips one byte of a
*copy* of `X3DH_INFO`, re-derives with the same IKM, and asserts the output diverges from the
committed vector — test-engineer independently verified this isn't tautological by actually mutating
the real label in `primitives.rs` and confirming `conformance.rs` fails as expected, then reverted.

**CI:** `.github/workflows/ci.yml`'s `test` job replaces the old `TODO: conformance vectors` line
with: regenerate (`cargo run -p xtask -- vectors`) → fail on any diff (`git diff --exit-code --
test-vectors/`) → run `cargo test -p meridian-crypto --test conformance`.

**Verification:** `cargo test -p meridian-crypto` (11 tests, all pass), `cargo test --workspace`
green, `cargo run -p xtask -- vectors` regenerates byte-identically across 5 repeated runs, `cargo
clippy -p meridian-crypto -p xtask -- -D warnings` clean, `cargo fmt --all -- --check` clean. No KDF
label, HKDF input/salt/info ordering, or wire byte layout changed anywhere — behavior is pinned, not
altered.

**Reviews:** **security-reviewer** PASS (confirmed no spec drift, visibility widening has no real
attack-surface increase since `test_support` is the only external path and is doc-hidden; one nit —
`#[doc(hidden)]` on `init_initiator_with_keypair` — applied). **test-engineer** PASS (confirmed the
conformance test and negative test are real gates, not tautologies; CI ordering correctly catches
accidental drift).

**Follow-up flagged, not fixed (out of scope):** test-engineer empirically demonstrated that a
*deliberate*, internally-consistent spec change — e.g. a real KDF label edit committed together with
regenerated vectors in the same PR — passes every CI layer here, since the generator, the
conformance test, and the negative test's own sanity baseline all call the same (now-wrong) real
derivation code and agree with each other. Closing this needs either an externally-sourced KAT
(explicitly out of this task's scope — browser/mobile consumers) or mandatory security-reviewer sign-
off enforced via CODEOWNERS on `apps/crypto/src/primitives.rs` and `test-vectors/**` — no CODEOWNERS
file exists anywhere in this repo yet. Recorded here for a future fix-task (adjacent to Group C's CI-
gates work, 1.8).
