**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire, F20) · **Depends on:** 1.15, 1.14 · **Status:** pending

## Goal
Feature 5's NAT/relay policy is proven at the wire level against the real backend, and the relay-only
privacy guarantee is enforced from **observed** gathered candidates rather than derived only from policy.

## Scope
- **In:**
  - Run the netns/tcpdump acceptance matrix (`tools/netns-nat-matrix.sh:71-78`) against the real
    `WebRtcTransport` from 1.15; exercise coturn for real (with the quota from 1.14); satisfy the spec's
    packet-capture criteria across the four NAT cells.
  - **F20:** classify actual gathered ICE candidates and abort the dial if a host/srflx candidate appears
    under relay-only, replacing the policy-derived privacy line at `apps/rendezvous/src/session.rs:296-299`
    with observation-based enforcement.
- **Out:** the backend itself (→ 1.15); media (Feature 10).

## Deliverables
1. Passing netns/tcpdump NAT matrix over the real backend (all four cells) exercising coturn.
2. Candidate-classification code that aborts a relay-only dial on any non-relay candidate + its test.
3. Updated `apps/rendezvous/src/session.rs` relay-only UX line derived from observed candidates.

## Risks / notes
This is the invariant that makes "relay-only" a real IP-hiding guarantee, not a label — the abort must be
**fail-closed** (any host/srflx candidate under relay-only aborts, never downgrades silently). Requires
1.15's backend and 1.14's coturn quota. If netns/tcpdump can't run in CI, gate it as an `--ignored` /
harness job and document how to run it.

## Tests

classification) · packet capture confirming relay-only emits only relay candidates.

## Reviews
