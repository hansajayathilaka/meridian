# 1.17 — ADR: deniability vs. envelope identity-key signature

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (on-the-fly decision) · **Depends on:** 1.1 · **Status:** pending

## Goal
Resolve the contradiction inside the design docs: the envelope signs every ciphertext with the identity
key (`Sign_IK{ratchet_ct}`, `docs/api/messaging-envelope-v1.md:96`) — third-party-provable authorship —
while threat-model goal 4 (`docs/security/threat-model.md:34`) claims deniability because identity-key
signatures are "confined to key distribution and signaling." Record an explicit, binding decision.

## Scope
- **In:** an ADR (via `/adr`, next number after 0015) that decides one of: (a) narrow the threat-model
  deniability claim to match the signed envelope, or (b) spec envelope v2 to drop the per-message
  identity-key signature (the ratchet AEAD already binds both identity keys). Update whichever doc the ADR
  chooses (`threat-model.md` and/or `messaging-envelope-v1.md`) to remove the contradiction.
- **Out:** implementing envelope v2 if that path is chosen (future build phase — record it as a follow-up).

## Deliverables
1. New ADR recording the deniability-vs-signature decision + rationale.
2. Edited `docs/security/threat-model.md` and/or `docs/api/messaging-envelope-v1.md` so the two agree.

## Risks / notes
This is a design decision, **not** a code fix — do not silently pick one; route to **architect** +
**security-reviewer**. If envelope v2 (drop the signature) is chosen, it is a wire change → versioned,
conformance-vectored (coordinate with 1.6), and deferred to a build phase, not done here. The docs must
not ship contradicting each other after this task.

## Tests
Docs CI link/mermaid check; grep confirming threat-model and envelope spec no longer contradict on
signing/deniability.

## Reviews
**architect** (ADR + wire-versioning implications), **security-reviewer** (deniability property).
