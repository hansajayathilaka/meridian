# 1.26 — Drive real `meridian` peers across the topology + capture pcaps

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire, prerequisite) · **Depends on:** 1.24, 1.25 · **Status:** pending

## Goal
Two real `meridian` processes, one in `ns-alice` and one in `ns-bob`, actually establish a session across
each of the four NAT cells via `ip netns exec`, using [1.24](./1.24-real-signaling-p2p-cli.md)'s
`session connect` command and [1.25](./1.25-netns-topology-coturn.md)'s topology, with `tcpdump`
capturing at `ns-bob` and `ns-turn` for every run.

## Scope
- **In:** wire `tools/netns-nat-matrix.sh`'s `cell()` to launch both peers (`ip netns exec ns-alice …
  session connect …` / `ip netns exec ns-bob …`) against 1.25's in-topology rendezvous, selecting the
  relay policy per cell (direct/prefer-relay/relay-only per the matrix's intent); start/stop `tcpdump` on
  `ns-bob` and `ns-turn` bracketing each run; save one pcap per cell per capture point with a clear,
  predictable naming scheme for [1.27](./1.27-pcap-assertions-ci.md) to consume; print the negotiated
  path (`session info`) per cell so a human can eyeball correctness before 1.27 automates it.
- **Out:** the strict pass/fail pcap assertions themselves and CI wiring (→
  [1.27](./1.27-pcap-assertions-ci.md)) — this task proves peers connect and pcaps exist, not yet that
  the pcaps prove the privacy/relay claims.

## Deliverables
1. All four cells produce a successful `session connect` across the real topology.
2. A pcap file per (cell × capture point) saved to a predictable path/naming scheme.

## Risks / notes
This is where 1.24's cross-process assumptions get exercised for real (real ICE gathering behind real
NAT, not localhost loopback-address ICE) — expect to find and fix small connectivity issues here; route
those through **connectivity-debugger**, not silent workarounds. If timeouts differ meaningfully from
localhost, note it but don't relax `dial_with_config`/`answer_with_config`'s existing bounded-wait
behavior to "fix" it — a real timeout under `udp-blocked` is a legitimate slow path, not a bug to paper
over.

## Tests
`sudo bash tools/netns-nat-matrix.sh cell <name>` for each of the four cells, asserting a successful
connect + non-empty pcaps (root/`NET_ADMIN` required).

## Reviews
**connectivity-debugger**, **test-engineer**.
