# 1.5 — Close the two zeroization gaps in meridian-crypto

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F5, F6) · **Depends on:** — · **Status:** done

## Goal
No secret material leaves scope un-zeroized in either the X3DH master-secret assembly or the
`DoubleRatchet` `Drop`, closing the two leaks that negate the surrounding `Zeroizing` discipline.

## Scope
- **In:**
  - **F5:** wrap the concatenated X3DH master secret at `apps/crypto/src/x3dh.rs:94-102,132-140` in
    `Zeroizing<Vec<u8>>` (or equivalent) before deriving the root key — today it is a plain `Vec<u8>`
    dropped without zeroization, negating the per-leg `Zeroizing` wrappers upstream.
  - **F6:** add the four header-encryption keys to `DoubleRatchet`'s `Drop` zeroize set at
    `apps/crypto/src/ratchet.rs:70-84`.
- **Out:** other crates; the broader `SecretStore` KDF change (→ 1.7).

## Deliverables
1. Edited `apps/crypto/src/x3dh.rs` (concatenation buffer wrapped).
2. Edited `apps/crypto/src/ratchet.rs` (`Drop` covers header-encryption keys).
3. Unit test(s) asserting zeroization coverage where observable.

## Risks / notes
Do not change KDF inputs/labels while wrapping the buffer — behavior must be byte-identical (the 1.6
vectors must still pass). `Zeroizing` must wrap the buffer that actually feeds the KDF, not a copy.

## Tests
`cargo nextest run -p meridian-crypto`

## Reviews
**security-reviewer** (zeroization completeness / FS discipline), **code-reviewer**.

## Result
**F5** — `apps/crypto/src/x3dh.rs`: the `ikm` concatenation buffer (`F ‖ DH1 ‖ DH2 ‖ DH3 ‖ [DH4]`) in
both `initiate()` and `respond()` is now bound as `Zeroizing<Vec<u8>>` instead of a plain `Vec<u8>`,
so it's wiped on drop after feeding `derive()`. No KDF input, label, or byte layout changed.

**F6** — `apps/crypto/src/ratchet.rs`: extracted the `Drop` body into `DoubleRatchet::zeroize_secrets`
(called by `Drop::drop`, so every drop path runs identical logic) and added zeroization of the four
previously-missed header-encryption keys `hks`, `hkr`, `nhks`, `nhkr`. Added unit test
`drop_zeroizes_all_secret_fields_including_header_keys`, which calls `zeroize_secrets()` directly
(avoiding UB from post-`Drop` memory inspection) against a struct literal with distinct non-zero
values in every secret field, including the four newly-covered ones.

Verified: `cargo test -p meridian-crypto` (6 tests, all pass, including all 5 pre-existing
forward-secrecy/PCS/persistence/out-of-order integration tests unaffected) and
`cargo clippy -p meridian-crypto -- -D warnings` clean. Reviewed by **security-reviewer** (PASS —
closes F5/F6 per the crypto CLAUDE.md key-hygiene rule, confirmed no KDF/derivation logic touched)
and **code-reviewer** (PASS — diff minimal and correctly scoped, test exercises the real Drop-invoked
code path with no drift risk).

Both reviewers independently flagged the same class of gap as out-of-scope follow-up, not required to
close F5/F6: intermediate `okm: [u8; N]` buffers in `primitives.rs` (`hkdf`, `kdf_rk`, `aead_seal`/
`aead_open`) hold derived secret material and aren't zeroized on scope exit; and `Skipped.hk` (the
per-skipped-message header key) is not zeroized alongside `Skipped.mk` in `zeroize_secrets()`. Neither
was named in this task's **In:** scope — recorded here for a future fix-task rather than silently
dropped.
