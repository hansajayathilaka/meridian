# 1.9 — Metrics-allowlist exhaustiveness test

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F14) · **Depends on:** — · **Status:** done

## Goal
The `/metrics` endpoint cannot leak a non-allowlisted metric family (e.g. a contact-graph metric) —
closing the vacuous-lint gap where a new family would pass CI today.

## Scope
- **In:** at `apps/rendezvous/tests/rendezvous.rs:293`, replace the presence-only assertion (allowlisted
  names are present) with **exhaustiveness**: every metric family actually rendered in the `/metrics` body
  must appear in `tools/metrics-allowlist.txt`, and no family outside the allowlist may appear. The
  metrics are hand-rendered in `apps/rendezvous/src/metrics.rs:4` (explicitly so the macro-form lint has
  nothing to flag), so the test must parse the rendered text body.
- **Out:** rewriting the hand-rendered metrics; the macro-form lint itself.

## Deliverables
1. Strengthened test in `apps/rendezvous/tests/rendezvous.rs` asserting rendered-body ⊆ allowlist and
   flagging any extra family.
2. Optional small render-introspection helper if needed to enumerate emitted families.

## Risks / notes
Parse metric **family** names from the rendered Prometheus text (strip labels/`_bucket`/`_sum`/`_count`
suffixes) so histograms don't false-positive. The test must fail if a new un-allowlisted family is added
— add a temporary family locally to confirm it trips, then revert.

## Tests
`cargo nextest run -p meridian-rendezvous`

## Reviews
**security-reviewer** (metadata-minimization invariant), **test-engineer** (exhaustiveness actually holds).

## Result
Strengthened `metrics_endpoint_exposes_allowlisted_names` in `apps/rendezvous/tests/rendezvous.rs`
with two helpers: `load_metrics_allowlist()` (reads `tools/metrics-allowlist.txt` at test time) and
`rendered_metric_families()` (parses the actual `/metrics` HTTP response body, skipping HELP/TYPE
comment lines, stripping label sets, and folding `_bucket`/`_sum`/`_count` suffixes back to their
base family). The new assertion checks `rendered.difference(&allowlist)` is empty — a one-way
subset check (the allowlist reserves names for not-yet-implemented metrics, so it's not a bijection)
that fails with the offending family name(s) if anything un-allowlisted is ever rendered. The
existing presence check (implemented families are actually rendered) is preserved so the
exhaustiveness check can't be trivially satisfied by rendering nothing.

**F14 closed:** a new non-allowlisted metric family (e.g. a contact-graph-derived counter) now fails
CI instead of passing silently.

**Negative-test verification:** temporarily added an un-allowlisted family
(`meridian_contact_graph_size_total`) to `metrics.rs`'s `render()`; the test failed with `metric
families rendered but not in tools/metrics-allowlist.txt: ["meridian_contact_graph_size_total"]`.
Fully reverted; `git diff --exit-code` clean afterward.

**Verification:** `cargo test -p meridian-rendezvous` → all tests pass; `cargo clippy --workspace
--all-targets -- -D warnings` and `cargo fmt --all -- --check` → clean.

**Reviews:** **security-reviewer** PASS (exhaustiveness check closes the metadata-minimization gap;
`tools/metrics-allowlist.txt` contents are all aggregate/operational, no per-user/contact-graph
shapes; noted as non-blocking that the check is family-name-only, not label-value-aware — moot today
since `metrics.rs` emits no labels). **test-engineer** PASS (independently reran the negative-test
cycle with the same result; noted a theoretical, not-currently-triggered edge case where a
legitimately `_count`-suffixed non-histogram counter could fold into an existing stem — no such name
exists today).
