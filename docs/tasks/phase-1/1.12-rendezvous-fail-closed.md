# 1.12 ‚Äî Rendezvous fail-closed config + feature-gate the tamper hook

**Phase:** 1 ‚Äî Review of Phase 0 ¬∑ **Feature:** review fix (F16, F17) ¬∑ **Depends on:** ‚Äî ¬∑ **Status:** done

## Goal
The server never silently weakens its posture on a config error, and the bundle-tamper test hook can
never ship in a release binary ‚Äî closing two threat-model goal 6 ("never silently weaker") violations.

## Scope
- **In:**
  - **F16:** at `apps/rendezvous/src/main.rs:32-38`, exit non-zero when a `--config` path is explicitly
    supplied but fails to parse ‚Äî today it warns and boots with defaults, silently turning invite-only
    into open registration. (No config supplied ‚Üí defaults is still fine.)
  - **F17:** gate the bundle-tamper test hook at `apps/rendezvous/src/ws.rs:230-235` behind a cargo
    feature excluded from release builds, instead of a runtime TOML flag present in the production binary.
- **Out:** broader config-schema validation work.

## Deliverables
1. Edited `apps/rendezvous/src/main.rs` ‚Äî fail-closed on explicit-but-unparseable config.
2. Edited `apps/rendezvous/src/ws.rs` + `apps/rendezvous/Cargo.toml` ‚Äî tamper hook behind a
   non-default, test-only cargo feature.
3. Tests: (a) bad explicit `--config` path exits non-zero; (b) tamper hook absent without the feature.

## Risks / notes
Distinguish "no config path given" (defaults OK) from "path given but unparseable/missing" (must fail).
The tamper-hook feature must not be in the default feature set and must be excluded from release builds
‚Äî confirm `cargo build --release` compiles the hook out entirely.

## Tests
`cargo nextest run -p meridian-rendezvous` ¬∑ `cargo build --release -p meridian-rendezvous` (hook absent).

## Reviews
**security-reviewer** (threat-model goal 6; hook cannot ship), **code-reviewer**.

## Result
**F16:** `apps/rendezvous/src/main.rs`'s `--config` argument is now `Option<String>`. `Some(path)`
(explicitly supplied) that fails to load/parse now prints a clear error and `std::process::exit(1)` ‚Äî
never boots with defaults. `None` (no flag given) still best-effort loads the conventional
`rendezvous.toml` from cwd, falling back to `Config::default()` silently if absent ‚Äî unchanged,
pre-existing behavior, and `Config::default()`'s `Admission::Open` is the honestly-documented default
posture, not a silently-weaker one.

**F17:** the bundle-tamper hook (`substitute_bundle` in `apps/rendezvous/src/auth.rs`, its call site in
`ws.rs::handle_fetch`) is now behind a non-default Cargo feature `test-tamper-hook`
(`apps/rendezvous/Cargo.toml`). `cargo build --release -p meridian-rendezvous` (default features)
compiles the hook out entirely (confirmed via `nm -C` ‚Äî `substitute_bundle` absent); with
`--features test-tamper-hook` it's present. The `allow_test_tamper` config bool stays always-present
(harmless plumbing, inert without the feature) and is documented as such in
`rendezvous.example.toml`. `apps/cli/Cargo.toml`'s dev-dependency on `meridian-rendezvous` and
`harnesses/mitm-sim/run.sh`'s `cargo test` invocation both needed `--features test-tamper-hook` added
‚Äî without it, the `--tamper` CLI demo test and the mitm-sim harness's substituted-bundle case would
have silently stopped testing anything (filter-matches-zero-tests exits 0). Both changes are scoped to
dev-dependencies/test invocations only, never the release binary.

**Verification:** `cargo test -p meridian-rendezvous` (both with and without `--features
test-tamper-hook`) clean; `cargo build --release -p meridian-rendezvous` (hook absent) and
`--features test-tamper-hook` (hook present) both clean; `cargo clippy --workspace --all-targets
-- -D warnings` (+ with the feature) and `cargo fmt --all -- --check` clean; `bash
harnesses/mitm-sim/run.sh` passes end-to-end.

**Reviews:** **security-reviewer** PASS (F16 control-flow correct, silent-defaults-fallback residual
on the `None` path is bounded/pre-existing/not a goal-6 violation, noted as a nit that it isn't
logged; F17 cfg-gating is structurally sound ‚Äî parse-time exclusion, stronger than a runtime check ‚Äî
`allow_test_tamper` correctly documented as inert without the feature). **code-reviewer** PASS
(control flow, cfg-gating consistency, new integration test's isolation/anti-flakiness, scope of the
cli/mitm-sim changes, and lint/fmt all verified independently).
