# 1.29 — ICE candidate-pair nomination stall under `direct`/`prefer-relay` (Bug A)

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire) · **Depends on:** 1.26 · **Status:** done

## Goal
Real `session connect` across a real multi-hop NAT currently never establishes under `IcePolicy::Direct`
or `PreferRelay` (`RTCIceTransportPolicy::All`) — it fails with `transport error: no candidate pair
selected yet`, even though `IcePolicy::RelayOnly` succeeds cleanly in the identical topology. This task
makes `direct`/`prefer-relay` actually work across a real NAT again, matching
[`seq-call-relay-fallback.mermaid`](../../architecture/diagrams/seq-call-relay-fallback.mermaid)'s
documented intent (direct/srflx fails, relay succeeds, within one ICE gathering phase).

## Scope
- **In:** root-cause-confirmed fix for the nomination stall. Try, in order (per architect review):
  1. A config-level fix in `apps/transport/src/webrtc_backend.rs::new_session` — tighten
     `SettingEngine::set_ice_timeouts` (disconnected/failed/keepalive) so permanently-unreachable
     host/srflx pairs transition to a terminal `Failed` state within `WAIT_TIMEOUT`, letting the
     *existing* `webrtc-ice` nomination logic fall through to the working relay-vs-relay pair on its
     own — no core-crate or design-doc change needed if this closes the gap.
  2. Only if (1) is proven insufficient: a session-level fallback in `apps/core/src/session.rs` (retry
     under `IcePolicy::RelayOnly` after `WAIT_TIMEOUT` elapses with no selected pair, when a TURN grant
     was available) — this is a real behavioral addition to the documented single-phase fallback model,
     so it REQUIRES updating `seq-call-relay-fallback.mermaid` and
     [Feature 5's spec](../../architecture/features/05-nat-traversal-relay-policy.md) (including
     surfacing the added latency in `session info`, per Feature 5's own "show the cost" risk note) and
     architect sign-off before merge — not a silent addition.
- **Out:** Bug B (TURN-over-TCP dependency gap under `udp-blocked` + `relay-only`) — that is
  [1.30](./1.30-turn-tcp-dependency-gap.md), a separate root cause. Any change to `IcePolicy`/`IceServer`
  type shapes or `relay::ice_config`'s signature (not needed for either fix path above).

## Deliverables
1. `full-cone` and `port-restricted` (policy `direct`) and `symmetric:symmetric` (policy `prefer-relay`)
   all connect successfully via `sudo bash tools/netns-nat-matrix.sh cell <name>` against the real rig.
2. Whichever fix path is taken, landed with its required reviews (config-level: connectivity-debugger
   confirms via live re-test; session-level: connectivity-debugger + architect + updated diagrams/spec).

## Risks / notes
Do not weaken/relax `WAIT_TIMEOUT` or any other bounded-wait as a "fix" — a genuinely slow-but-working
path is a legitimate finding, not a bug to paper over; this task is about a permanent stall, not slowness.
If a session-level retry fallback is ultimately needed, it must not silently double session-establishment
latency without surfacing that cost (Feature 5's spec already flags this as something the demo must show).

## Tests
`sudo bash tools/netns-nat-matrix.sh cell full-cone` / `cell port-restricted` / `cell symmetric:symmetric`
— each must report `[nat-matrix] cell='...': ALL CHECKS PASSED` with `session connect` actually
establishing on both sides (not just the topology/coturn/tcpdump mechanics, which already pass).

## Reviews
**connectivity-debugger** (confirms the fix against the real rig), **architect** (only if the session-level
fallback path is taken — design-doc/diagram consistency).

## Decision (recorded — Step 1 tried, proved insufficient; Step 2 shipped, architect-signed-off)
Step 1 (`SettingEngine::set_ice_timeouts` in `apps/transport/src/webrtc_backend.rs::new()`, tightened to
`ICE_DISCONNECTED_TIMEOUT=2s`/`ICE_FAILED_TIMEOUT=4s`/`ICE_KEEPALIVE_INTERVAL=2s`) was tried first, exactly
per the ordering above, and kept as a genuine improvement — but live re-testing against the real rig with
`webrtc_ice=trace` logging and coturn verbose logging showed even the relay-vs-relay pair (which connects
instantly under `RelayOnly`) never got a single successful STUN response under `Direct`/`PreferRelay`,
timing out via `webrtc-ice`'s fixed, non-configurable `max_binding_requests`(7)×`check_interval`(200ms)≈1.4s
budget regardless of timeout tuning — a real non-convergence, not a timing artifact (a "thundering herd
from batched `add_ice_candidate` calls" hypothesis was also tested and disproved: pacing candidate delivery
by 250ms still produced zero responses for the relay pair). Step 1 alone was therefore proven insufficient,
justifying Step 2 per the pre-approved ordering.

**Step 2 shipped**: `apps/core/src/session.rs`'s `dial_with_config`/`answer_with_config` now retry the
entire offer/answer/ICE exchange once, forcing `IcePolicy::RelayOnly`, when the first attempt fails with
`TransportError::NoPath` under `Direct`/`PreferRelay` **and** a TURN grant was available
(`relay_fallback_eligible`). The retry is a full second round trip through the signaling relay (not a
silent instant retry); its cost is surfaced via new `SessionInfo::relay_fallback`/`relay_fallback_wait_ms`
fields (shown in `session info`'s `Display`, `session connect --json`, and `doctor`), never hidden.
`docs/architecture/diagrams/seq-call-relay-fallback.mermaid` and
[Feature 5's spec](../../architecture/features/05-nat-traversal-relay-policy.md) were updated in lockstep
with the code to document the new two-phase model.

**Live-rig verification** (independently re-run, not just the implementing agent's self-report):
`sudo bash tools/netns-nat-matrix.sh cell full-cone`, `cell port-restricted`, and `cell symmetric:symmetric`
all report `ALL CHECKS PASSED` with `path=relay` and `relay_fallback:true` in the JSON output, confirmed
reliably repeatable across multiple fresh runs (one unrelated, pre-existing rig race — reusing the same
persistent test identities across repeated runs occasionally hits a stale-bundle/prekey error — required a
retry with a fresh rig teardown, unrelated to this fix). `cargo fmt --check`, `cargo clippy --workspace
--features webrtc --all-targets -- -D warnings`, and the full `meridian-core`/`meridian-transport`/
`meridian-cli` test suites (with `--features webrtc`) all pass with no regressions.

**Architect sign-off** (2026-07-23): reviewed `session.rs`'s retry logic, `chat.rs`'s ratchet-reuse
guarantee (confirmed the second `recv_sdp`/`open_bytes` on retry is an ordinary ratchet-advance, not a
second X3DH open, since `ChatState::open_bytes` only calls `Session::respond` when no session exists yet),
and the updated diagram/spec — signed off with no correctness or architectural concerns. See
[1.26](./1.26-netns-drive-and-capture.md)'s Status for the same sign-off's resolution of the related
"all four cells must connect" completion-criteria question.
