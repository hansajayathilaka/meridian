# 1.23 — NAT/relay wire-level acceptance matrix (netns/tcpdump)

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire-level) · **Depends on:** 1.22, 1.14 · **Status:** split (see below)

## Split before implementation
Investigating this task (before any of its Scope was implemented) found its own Scope line rested on a
false assumption: "drive two real meridian peers across ns-alice/ns-bob using 1.22's `--transport webrtc`
flag." 1.22 (correctly, per its own risk note) implemented that flag only on `meridian session demo`,
which runs **both** simulated peers inside **one** process over an in-process `LoopbackFabric`/`MemRelay`
— there is no way for two separate OS processes to rendezvous over it, and no CLI command or core-layer
plumbing yet lets one process dial/answer a P2P session via the **real** rendezvous. That gap, plus the
combined scale of the netns topology + coturn orchestration + peer-driving + pcap analysis + CI wiring
(comparable to building a new feature end-to-end plus a full test harness), is too large for one PR — the
same situation that caused this task to be split out of 1.16 in the first place.

Split into four sub-tasks, in dependency order:
- [1.24](./1.24-real-signaling-p2p-cli.md) — real-signaling `SignalRelay` + `meridian session connect`
  CLI command (the missing prerequisite this task's Scope assumed already existed).
- [1.25](./1.25-netns-topology-coturn.md) — netns topology + NAT-flavor emulation + coturn/rendezvous
  orchestration (independent of 1.24; depends only on 1.14).
- [1.26](./1.26-netns-drive-and-capture.md) — drive real peers across the topology + capture pcaps
  (depends on 1.24 + 1.25).
- [1.27](./1.27-pcap-assertions-ci.md) — pcap-analysis assertions + CI/harness wiring; **this is the task
  that actually closes finding F11's wire-level half**, not this one.

A fifth item was flagged during the split but is **not** part of F11's closure: the "active relay-rewrite"
adversarial test noted in `apps/core/src/session.rs`'s module doc and `harnesses/mitm-sim/run.sh` (both
currently citing this task's number) is tracked separately as
[1.28](./1.28-active-relay-rewrite-test.md) — a distinct, security-sensitive server-side change, not
folded into 1.24 to keep that task mergeable on its own.

Nothing in this task's original Scope/Deliverables below was implemented under *this* task number; it is
superseded in full by 1.24–1.27, kept below only as the historical record of what the sub-tasks replace.

## Goal (superseded — see split above)
Feature 5's NAT/relay policy is proven at the wire level against the real backend: coturn actually
relays traffic, packet captures confirm the four NAT-matrix cells behave as specified, and — the load-
bearing privacy claim — a capture at the peer under `relay-only` contains zero of the dialing side's
host/srflx addresses.

## Scope
- **In:** implement the real body of `tools/netns-nat-matrix.sh`'s `cell()` function (today a stub that
  only calls `meridian doctor`'s in-process simulation) — build the six-namespace topology
  (`ns-alice`/`ns-natA`/`ns-net`/`ns-turn`/`ns-natB`/`ns-bob`), apply the NAT flavor per cell (full-cone,
  port-restricted, symmetric via random-fully/per-connection source ports, `--block-udp` for
  udp-blocked), launch coturn in `ns-turn` with real ephemeral HMAC credentials matching 1.14's
  `user-quota` scheme, drive two real `meridian` peers across `ns-alice`/`ns-bob` using 1.22's
  `--transport webrtc` flag, and assert with tcpdump/pcap analysis that: (a) the negotiated path matches
  the expected rung per cell; (b) under `relay-only`, a capture on `ns-bob` carries zero packets
  containing alice's host/srflx addresses; (c) a capture on `ns-turn` sees only DTLS ciphertext, never
  plaintext.
- **Out:** the CLI transport flag itself (→ 1.22, a hard dependency); any change to `WebRtcTransport`,
  `enforce_relay_only`, or other code already landed by 1.15/1.16 — this task is proof, not new
  candidate-handling logic; media (Feature 10).

## Deliverables
1. Passing netns/tcpdump NAT matrix over the real backend (all four cells) exercising coturn for real.
2. pcap-analysis assertions (script or small Rust/Python helper) proving criteria (a)-(c) above, not just
   "the demo didn't crash."
3. CI wiring if root+netns is available there, or — per the original risk note — an `--ignored`/harness
   job gated on root/`NET_ADMIN` with clear SKIP messaging and documented manual run instructions if not
   (mirror `tools/netns-nat-matrix.sh`'s existing `need_root()` graceful-skip pattern).

## Risks / notes
This is the invariant that makes "relay-only" a real IP-hiding guarantee provable on the wire, not just
in-process — the pcap assertions must be strict (any host/srflx packet from the dialing side under
relay-only is a hard failure, not a warning). Requires 1.22 (CLI transport selection) and 1.14 (coturn
quota) to already be in place. `apps/core`'s own `enforce_relay_only` (landed in 1.16) is a second,
independent layer of the same guarantee — this task proves it end-to-end at the network level, it does
not re-implement it. Building and tearing down netns/veth topology plus coturn config is real
system-level surface; keep teardown (`tools/netns-nat-matrix.sh down`) robust against a failed run
leaving stale namespaces behind.

## Tests
`bash tools/netns-nat-matrix.sh matrix` (all four cells, root/`NET_ADMIN` required) · the pcap-assertion
helper from deliverable 2.

## Reviews
**connectivity-debugger** (topology + NAT flavors + ICE behavior), **security-reviewer** (the
relay-only zero-leak pcap assertion, fail-closed), **devops** (CI wiring / gating), **test-engineer**.
