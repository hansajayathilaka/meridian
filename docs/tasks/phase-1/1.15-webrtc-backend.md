## Status (done)
`WebRtcTransport` (`apps/transport/src/webrtc_backend.rs`) implements the full `Transport` trait over
webrtc-rs 0.17: real ICE gathering, `RTCIceTransportPolicy::Relay` for a genuine `relay-only` (never
gathers host/srflx), pre-negotiated SCTP data channels (avoids the in-band duplicate-channel footgun â€”
see the module's doc comment for the full design writeup), and fingerprint binding read from the
ratchet-encrypted-envelope-carried SDP text (real WebRTC/DTLS enforces cert-matches-SDP-fingerprint as
a hard precondition of the handshake ever completing â€” proven by
`apps/transport/tests/webrtc_backend.rs::tampered_remote_fingerprint_never_connects`). `meridian-core`
forwards a matching `webrtc` feature; `dial_with_config`/`answer_with_config` now close the transport
session on any failed setup path (a real backend leaks a live `RTCPeerConnection`, not just a HashMap
entry, if it doesn't) and fail fast on a connection that never converges instead of hanging forever on
an unbounded `recv()`.

  restart does not (yet) fulfill the resumability promise" section. Calling webrtc-rs's real
  ICE-agent restart on an already-connected `RTCPeerConnection` with no peer-side signaling breaks
  the live connection (verified empirically); there is no ctrl-channel renegotiation path today to
  carry a restart offer to the peer. This means system-design Â§5.2/Â§7.3's "resumable... ICE restarts

- **relay-transport classification (udp/tcp/tls-443) is not derivable** from webrtc-rs 0.17.1's stats
