# 1.30 — TURN-over-TCP client gap under `relay-only` + `udp-blocked` (Bug B)

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire) · **Depends on:** 1.26 · **Status:** pending

## Goal
Real `session connect` under `IcePolicy::RelayOnly` hangs completely (no output, killed only by an outer
script timeout) against the `udp-blocked` NAT cell. Root-caused by connectivity-debugger via direct source
read: the pinned `webrtc-ice` 0.17.1's `agent_gather.rs::gather_candidates_relay` only implements
`transport=udp` TURN gathering — `transport=tcp`/`turns:` URLs hit an explicit `else` branch (with a `TODO`
comment acknowledging the gap) and are silently dropped. **No wiring change in
`apps/cli/src/session_connect.rs` or `apps/transport/src/webrtc_backend.rs` can fix this** — the code path
that would handle a TCP TURN URL does not exist in this dependency version. This task resolves that gap,
one way or another, honestly.

## Scope
- **In:**
  1. Check whether a semver-compatible `webrtc`/`webrtc-ice` release adds client-side TURN-over-TCP
     support. Per architect review, `webrtc-ice`/`webrtc-rs` are NOT on
     [stack.md](../../architecture/stack.md)'s wire-critical pin list (unlike `ciborium`/`ed25519-dalek`/
     the ratchet lib) — a routine version bump is `rust-dev`'s call, no ADR needed, provided it's
     semver-compatible and passes the existing test suite + `cargo-deny`.
  2. If no upstream fix exists yet: honestly document `udp-blocked + relay-only` as a currently-unsupported
     real-backend combination. Extend
     [Feature 5's spec](../../architecture/features/05-nat-traversal-relay-policy.md)'s "Verification
     status" note (added at 1.16) — it currently reads as "simulation-only, not yet proven against a real
     backend," which is weaker than what's now actually known: it is **proven impossible** at the pinned
     dependency version, not merely unverified. Point the note at this task (or its resolution).
  3. Either way, also add a bounded overall timeout around the whole gather-and-connect flow in
     `apps/transport/src/webrtc_backend.rs` — connectivity-debugger found the current hang exceeds the
     existing 15s `WAIT_TIMEOUT` entirely (it isn't bounding this failure mode at all), so at minimum this
     should fail loud within a few seconds instead of hanging silently past 90s, regardless of whether the
     dependency bump lands.
- **Out:** Bug A (ICE nomination stall under `direct`/`prefer-relay`) — that's
  [1.29](./1.29-ice-nomination-relay-fallback.md), a separate root cause. Re-opening
  [1.14](./1.14-feature5-honesty.md) itself (done) — this is a new finding, tracked here, not a reason to
  reopen a completed task.

## Deliverables
1. A decision, recorded here: dependency bump (with the version and what changed) OR documented
   limitation (with the exact spec/doc wording change).
2. If bumped: `udp-blocked` (policy `relay-only`) connects successfully via
   `sudo bash tools/netns-nat-matrix.sh cell udp-blocked` against the real rig.
3. If documented: Feature 5's spec wording updated to state the real-backend limitation plainly (not
   "not yet proven" but "currently unsupported, see 1.30"), and `apps/cli/tests/nat_relay.rs`'s
   `udp_blocked_falls_back_to_tls_443` in-process test's doc comment updated to note it verifies the
   *simulated* enum-based matrix only, not the real webrtc-ice-backed client.
4. Either way: a bounded timeout around the gather-and-connect flow in `webrtc_backend.rs` so this failure
   mode fails loud within seconds, not hangs past 90s.

## Risks / notes
Don't silently downgrade `relay-only`'s guarantee to make this pass — if the dependency bump doesn't land,
the honest path is documenting the gap, not quietly relaxing `udp-blocked`'s NAT-flavor definition or
`relay-only`'s semantics. A dependency bump big enough to require API changes in
`apps/transport/src/webrtc_backend.rs` is still in scope here; a dependency bump that turns out to need an
architecture-level rethink (e.g. switching ICE-agent libraries entirely) is NOT — escalate that to
`architect` rather than attempting it under this task.

## Tests
`sudo bash tools/netns-nat-matrix.sh cell udp-blocked` (must pass if bumped) · `cargo test -p meridian-cli
--test nat_relay` (doc-comment wording, if documented-limitation path taken) · a new bounded-timeout
regression test in `meridian-transport` proving the gather-and-connect flow fails within a few seconds
under a TURN-TCP-only, UDP-blocked scenario (either path).

## Decision (recorded — documented-limitation path taken; no dependency bump)
Checked crates.io for a newer `webrtc-ice` release: `0.17.2` exists (published after `0.17.1`, a
semver-compatible patch bump). Downloaded and diffed both crates' source directly — `0.17.2` only
changes ICE consent-freshness bookkeeping (`agent_internal.rs`) and a `Candidate::close()` ordering
fix (`candidate_base.rs`); `agent_gather.rs::gather_candidates_relay` is **byte-identical** to
`0.17.1`, including the same `else` branch and the same upstream `TODO` for `transport=tcp`/`turns:`
URLs. `max_version`/`newest_version` for `webrtc-ice` on crates.io is `0.17.2` — there is no later
release at all, so no bump (semver-compatible or otherwise) closes this gap today. Per this task's own
disposition, that rules out the bump path; took the documented-limitation path instead:

1. [Feature 5's spec](../../architecture/features/05-nat-traversal-relay-policy.md) now states plainly
   that `udp-blocked + relay-only` is **proven impossible** at the pinned dependency version (not
   "not yet proven"), with the direct-source-read evidence and a pointer to this task.
2. [`apps/cli/tests/nat_relay.rs`](../../../apps/cli/tests/nat_relay.rs)'s
   `udp_blocked_falls_back_to_tls_443` doc comment now says explicitly that it exercises only the
   simulated, in-process `NatScenario`/`LoopbackTransport` matrix, not the real `webrtc-ice`-backed
   client (which cannot achieve this path at all). No assertion/behavior in that test changed.
3. `apps/transport/src/webrtc_backend.rs::local_candidates` now wraps the whole gather flow
   (`ensure_committed` — which is what actually kicks off ICE/TURN gathering for the dialer — plus
   the wait for it to finish) in one outer `GATHER_TIMEOUT` (20s), rather than only the last
   "wait for `gather_done`" step. Verified against the real six-namespace rig:
   `sudo bash tools/netns-nat-matrix.sh cell udp-blocked` (policy `relay-only`) now completes in
   ~34s total (well inside the harness's existing 90s outer timeout) and fails with a clear
   `transport error: no candidate pair selected yet` on both peers, instead of hanging with no
   output. A hermetic regression test,
   `relay_only_gathering_fails_fast_against_a_silent_turn_server` in
   `apps/transport/tests/webrtc_backend.rs`, reproduces the shape of the failure (a TURN server
   whose UDP path never answers, under `IcePolicy::RelayOnly`) without needing root/netns, and
   asserts `local_candidates` returns well inside a bounded window rather than creeping toward an
   outer test guard.

The underlying connection still cannot succeed under this combination — that requires either a
`webrtc-ice`/`webrtc-rs` release with client TURN/TCP support (none exists as of this check) or an
ICE-agent library change (architect territory, not attempted here) — but the failure mode is now
fast and loud, not a silent multi-minute hang.
