# 1.24 тАФ Real-signaling `SignalRelay` + `meridian session connect` CLI command

**Phase:** 1 тАФ Review of Phase 0 ┬╖ **Feature:** review fix (F11 wire, prerequisite) ┬╖ **Depends on:** 1.22 ┬╖ **Status:** done

## Goal
One `meridian` process can dial or answer a real P2P session (`meridian_core::session::dial_with_config`/
`answer_with_config`) against a peer over the **real** rendezvous тАФ closing the gap that today only
`MemRelay` (in-process channel pair) implements `SignalRelay`. This is what lets two separate OS
processes (in two separate network namespaces, [1.26](./1.26-netns-drive-and-capture.md)) actually
establish a session, which `session demo`'s single-process simulation can never do (`LoopbackFabric` is
in-process only).

## Scope
- **In:** a `SignalRelay` impl in `apps/core` wrapping the real `SignalingClient`
  (`apps/signaling/src/client.rs`) тАФ `route()` тЖТ `send()`, `next_deliver()` тЖТ `recv()`, filtering
  delivered blobs on the expected peer `ik` (mirroring the existing `recv_sdp`-style filtering already
  used elsewhere in signaling). Lives in `apps/core` (a new module, e.g. `apps/core/src/signal_relay.rs`)
  because `meridian-core` already depends on `meridian-signaling`, and `meridian-signaling` does not
  depend back on `meridian-core` тАФ implementing `SignalRelay` (defined in `apps/core/src/session.rs`)
  inside `apps/signaling` would create a dependency cycle.
- A new CLI subcommand (name is rust-dev's call, e.g. `meridian session connect <peer-id> --server <url>
  --transport webrtc`), mirroring `apps/cli/src/chat.rs`'s connect/publish-bundle/fetch-peer-bundle/
  role-by-key-order setup, but calling `dial_with_config`/`answer_with_config` instead of relaying chat
  envelopes. The signaling `SignalingClient` connection is dropped once the P2P session is established
  (T04's "servers out of the data path" property, now over a real socket, not just in-process).
  `--transport loopback` is **rejected, not silently substituted** on this command тАФ `LoopbackTransport`'s
  fabric is in-process only, so a real cross-process command has no working loopback mode; fail closed
  with a clear message, mirroring 1.22's existing fail-closed idiom for invalid flag combinations.
- A real-process integration test: two compiled `meridian` binaries, spawned by the test as actual child
  processes, against a real (test-local) `meridian-rendezvous` instance on localhost, using
  `--transport webrtc`.
- **Out:** any NAT/namespace topology, coturn, tcpdump (тЖТ [1.26](./1.26-netns-drive-and-capture.md)); any
  change to `WebRtcTransport`, `Transport` trait, or `enforce_relay_only` (already landed, 1.15/1.16);
  wiring P2P into `meridian chat` itself (still a distinct future feature, per 1.22's precedent). The
  "active relay-rewrite" adversarial test (rendezvous `test-tamper-hook` rewriting *routed* blobs, not
  just `Fetch` bundles) is a separate, security-sensitive server-side change тАФ tracked as
  [1.28](./1.28-active-relay-rewrite-test.md), not folded in here.

## Deliverables
1. `SignalRelay`-over-`SignalingClient` adapter in `apps/core` + unit tests.
2. The `session connect` CLI command + fail-closed `--transport loopback` rejection.
3. A two-real-process localhost integration test asserting `transport=webrtc-datachannel` in
   `session info` and a chat exchange, driven through a real rendezvous instance (not `MemRelay`).

## Risks / notes
No new wire type or `Op` is needed тАФ `SignalingClient::route`/`Deliver{from, blob}` already carries
arbitrary opaque envelope bytes; the SDP offer/answer/ctrl content rides the identical wire path T03
chat already uses (`docs/architecture/features/04-p2p-session-substrate.md` ┬з7.1 already specifies this
тАФ filling it in is not new design, so **no ADR is required**). This is nonetheless the first production
code path actually binding real rendezvous signaling to P2P session establishment (previously only
exercised via `MemRelay` in tests/demo) тАФ recommend a lightweight **architect** sanity read before merge.

## Status (done)
`apps/core/src/signal_relay.rs` (new) adds `RendezvousRelay`, a `SignalRelay` impl borrowing a connected
`SignalingClient`: `send` maps `route()`'s `Ok(false)` (peer not currently connected) to a hard
`SessionError::Relay` (no mailbox story for the offer/answer exchange itself тАФ an offline peer must
abort, not silently continue), and any `SignalError` to `SessionError::Relay(e.to_string())`; `recv`
forwards `next_deliver()`'s `(from, blob)` verbatim, since `session::recv_sdp` already filters by peer.
Five unit tests cover both mappings. `apps/cli/src/session_connect.rs` (new) adds
`meridian session connect <peer-id> --server <url> --transport <loopback|webrtc>`, mirroring
`chat.rs`'s connect/publish-bundle/fetch-peer-bundle/role-by-key-order setup but calling
`dial_with_config`/`answer_with_config` over a `RendezvousRelay` instead of relaying chat envelopes;
`--transport loopback` is rejected outright (no cross-process loopback fabric), and тАФ found during
security review, see below тАФ a configured relay policy other than `direct` for the peer is also rejected
outright, since no real TURN relay exists yet (that's 1.25/1.27) and silently connecting `direct`
regardless would defeat `config set policy relay-only`. Both rejections are covered by
`apps/cli/tests/session_connect.rs` (new, unconditional тАФ mirrors 1.22's
`session_demo_rejects_webrtc_transport_without_the_feature` precedent for the no-feature path) and
`apps/cli/tests/session_connect_webrtc.rs`'s `refuses_to_connect_when_the_configured_policy_for_the_peer_is_not_direct`.

The two-real-process integration test (`session_connect_webrtc.rs`) initially hung (30s timeout, one
side stuck with no output past its first bundle-fetch retry). A connectivity-debugger investigation
found the actual bug was **not** in signaling/ICE at all: `WebRtcTransport::close()`
(`apps/transport/src/webrtc_backend.rs`) called `pc.close()` immediately after the caller's last
`send()`, with no wait for the data channel's SCTP send buffer to actually flush тАФ `RTCDataChannel::send`
returning only means the bytes were accepted into that buffer, not that they left the process. A message
queued just before `close()` (the responder's final chat reply, in this command's flow) could be
silently dropped, leaving the initiator blocked forever on an unbounded `recv()`. Invisible to every
prior test: the single-process gated suite never calls `close()` right after a `send()`, and
`LoopbackTransport::send` delivers synchronously into the peer's inbox before returning, so a
same-process `close()` can never race a delivery in flight тАФ the race is specific to a real async
transport with real OS I/O. Fixed by draining each data channel's `buffered_amount()` to zero (bounded
by a new 500ms `CLOSE_DRAIN_TIMEOUT`, so a genuinely-gone peer can't hang teardown forever) before
`pc.close()`. A new regression test, `apps/transport/tests/webrtc_backend.rs`'s
`send_immediately_before_close_is_still_delivered`, pins this тАФ manually verified to fail reliably (3/3
runs) with the drain loop reverted, and pass with it restored.

Reviewed by **test-engineer**, **security-reviewer**, and **architect** тАФ all clean, no blocking
findings after fixes. security-reviewer found one **Medium** finding (`session connect` hardcoded
`IcePolicy::Direct` and never resolved the user's configured relay policy, silently exposing host/srflx
candidates to the peer even under a configured `relay-only`) тАФ fixed by adding the fail-closed policy
check described above, plus its regression test, both landed before this task closed. security-reviewer
also confirmed: the `Ok(false)`-is-hard-error mapping creates no new oracle (the rendezvous already knows
who's connected); envelope opacity is genuinely unbroken (`chat.seal_bytes`/`open_bytes` wrap every
`SignalContent` on both the send and receive path, no bypass); the ┬з4.6 fingerprint check and F20
relay-only enforcement are exercised unmodified through the shared core `dial_with_config`/
`answer_with_config` path, not reimplemented in the CLI; and bundle-publish-before-role-check ordering is
genuinely equivalent to `chat.rs`'s established pattern (no new leak to the rendezvous). Two low-severity,
non-blocking notes were recorded rather than dropped: (1) real-socket opacity for `SignalContent` is
proven by code inspection and the existing `MemRelay`-path opacity audit, not yet by a wire-capture
harness against the real socket this task adds тАФ tracked under 1.27/1.28, not closed here; (2) a freshly
published bundle's `spk_secret`/OTK secrets are never persisted to disk by `session connect` (unlike
`chat.rs`'s `save_state`), so a bundle can be orphaned if fetched by a third party after the process
exits тАФ an availability/self-DoS edge case, no confidentiality impact, left as a known limitation of this
demo-scale command. test-engineer separately noted the `CLOSE_DRAIN_TIMEOUT` fix is a bounded mitigation,
not a delivery proof (a genuinely lost-in-flight packet after leaving the local SCTP queue but before
peer ack is still possible, just narrowed) тАФ accepted as the right tradeoff (an unbounded drain could
hang teardown forever against a vanished peer) and now has the regression test described above; it also
flagged that `--json` mode's chat-exchange lines are computed but never printed (mirroring
`session.rs`'s pre-existing `run_demo_generic` pattern), so the integration test's content-transfer proof
is currently weaker than its "established" proof тАФ recorded as a known, non-blocking gap rather than
fixed in this task (out of scope: touches `session.rs`'s shared printing convention, not just this
command). architect confirmed no ADR is needed (the design was already specified in
`docs/architecture/features/04-p2p-session-substrate.md` ┬з7.1), the dependency direction is correct
(`signal_relay.rs` lives in `apps/core`, not `apps/signaling`), and the server/core boundary is
untouched; it flagged one non-blocking product-completeness note (today's fail-closed "no mailbox for
first contact" behavior should stay tracked as a T07 follow-up so `session connect` isn't permanently
unable to reach an offline first-contact peer once mailbox lands).

All tests pass: `cargo test -p meridian-core -p meridian-transport --features webrtc` (10+4+4+3+8+6+5=40
tests across the touched crates, zero regressions from the `close()` change), `cargo test -p meridian-cli`
(non-feature, 8 test files) and `cargo test -p meridian-cli --features webrtc` (9 test files including
both new `session_connect*` files), each run multiple times for stability given the fix's own history.
`cargo clippy --all-targets -D warnings` clean on every default/`webrtc` combination across
`meridian-core`/`meridian-transport`/`meridian-cli`; `cargo fmt --check` clean; `cargo build --workspace`
clean both with and without `--features meridian-cli/webrtc`.
