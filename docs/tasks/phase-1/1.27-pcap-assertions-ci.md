# 1.27 — pcap-analysis assertions + CI/harness wiring (closes F11 wire-level)

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire-level) · **Depends on:** 1.26 · **Status:** pending

## Goal
The pcap captures from [1.26](./1.26-netns-drive-and-capture.md) are turned into strict, fail-closed
pass/fail assertions — this is the task that actually closes F11's wire-level half, not merely "the demo
didn't crash." This is the invariant that makes "relay-only" a real IP-hiding guarantee provable on the
wire, not just in-process (1.16's `enforce_relay_only`) or at the CLI-plumbing level (1.24).

- **In:** a pcap-analysis helper (script or small Rust/Python tool) asserting: (a) the negotiated path
  per cell matches the expected rung (direct/srflx/relay-udp/relay-tls-443); (b) under `relay-only`,
  **zero** packets in `ns-bob`'s capture contain alice's host/srflx addresses — any match is a hard
  failure, never a warning; (c) `ns-turn`'s capture contains only DTLS ciphertext, never plaintext (e.g.
  no cleartext SDP/chat bytes visible via a simple string/pattern scan); wire
  `tools/netns-nat-matrix.sh matrix` to run all four cells end-to-end (1.25 topology → 1.26 drive+capture
  → this task's analysis) and exit non-zero on any assertion failure; CI wiring if root+netns is
  available there, else an `--ignored`/harness job gated on root/`NET_ADMIN` with clear SKIP messaging
  and documented manual run instructions, mirroring `tools/netns-nat-matrix.sh`'s existing `need_root()`
  pattern.
- **Per architect review (2026-07-23, following 1.26/1.29/1.30's resolution)**: split assertions (a)-(c)
  by what each presupposes rather than treating all four cells uniformly. Assertions (a) and (c) both
  presuppose an established path/data flow — scope these strictly to the 3 real-connecting cells
  (`full-cone`, `port-restricted`, `symmetric:symmetric`); `udp-blocked` has no negotiated rung and no
  TURN-relayed application data to assert about (1.30: proven-impossible dependency gap, not a bug this
  task re-proves). Assertion (b) (the `relay-only` zero-host/srflx-leak invariant) is the anonymity-model
  "must never" guarantee and should still run against `udp-blocked`'s pcap too — it must hold trivially
  there (1.16's `enforce_relay_only` means host/srflx are never gathered regardless of whether the
  session ultimately connects), and verifying the zero-leak property holds even during a failed
  `relay-only` attempt is a stronger proof, not one to skip. Add one narrow assertion specific to
  `udp-blocked`: the capture shows no established data channel/successful candidate-pair selection, and
  the process exit + error text match 1.30's documented, bounded failure mode (fails fast, not a hang) —
  proving the "fails fast and loud" claim on the wire, not just via CLI stdout.
- **Out:** any new topology/driving logic (already done in 1.25/1.26); any change to `WebRtcTransport` or
  `enforce_relay_only` — this task proves the existing 1.16 guarantee at the wire level, it does not
  re-implement it. Making `udp-blocked` actually connect (that's 1.30's territory, already resolved as a
  documented dependency limitation — not re-litigated here).

## Deliverables
1. The pcap-analysis helper with strict (a)-(c) assertions, split per the scope note above (3 cells get
   the full (a)-(c) treatment; `udp-blocked` gets (b) plus the fails-fast-on-the-wire assertion).
2. `bash tools/netns-nat-matrix.sh matrix` passing its now-amended acceptance bar end-to-end (3/4 cells
   connect and pass all assertions; `udp-blocked` fails fast with the documented 1.30 error and passes
   its own narrower assertions) — not "all four connect," per 1.26's architect-approved amendment.
3. CI wiring or a documented gated/manual harness job.
