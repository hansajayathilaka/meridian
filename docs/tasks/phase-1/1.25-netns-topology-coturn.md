# 1.25 — netns topology + NAT-flavor emulation + coturn/rendezvous orchestration

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire, prerequisite) · **Depends on:** 1.14 · **Status:** done

## Goal
The six-namespace topology (`ns-alice`/`ns-natA`/`ns-net`/`ns-turn`/`ns-natB`/`ns-bob`) exists and is
provably correct as *networking infrastructure* — real NAT behavior, real coturn, real rendezvous —
before any Meridian process is driven across it ([1.26](./1.26-netns-drive-and-capture.md)'s job).
Independent of [1.24](./1.24-real-signaling-p2p-cli.md); only depends on 1.14 (coturn user-quota), so it
can proceed in parallel with 1.24.

## Scope
- **In:** build the real veth/bridge topology in `tools/netns-nat-matrix.sh` (reusing/extending
  `tools/netns-two-lans.sh`'s `link_pair`/`nat_router` helpers where they fit); apply the per-cell NAT
  flavor on `ns-natA`/`ns-natB` — full-cone (plain MASQUERADE), port-restricted (conntrack-restricted
  reply rules), symmetric (`--random-fully`/per-connection source ports), udp-blocked (drop UDP egress so
  only TCP/TLS survive); launch real coturn in `ns-turn`; launch a real `meridian-rendezvous` instance
  reachable from both `ns-alice`/`ns-bob` so ephemeral HMAC TURN credentials are minted through the real
  `TurnReq`/`TurnGrant` flow (`apps/signaling/src/client.rs::request_turn_credentials`) matching 1.14's
  `user-quota=4` scheme — not a hand-rolled coturn-only credential script; robust `down` teardown covering
  every new resource (iptables rules, coturn/rendezvous processes), safe against a failed run leaving
  stale namespaces behind.
- **Out:** driving any `meridian` peer process, tcpdump capture, pcap analysis, CI wiring (→
  [1.26](./1.26-netns-drive-and-capture.md)/[1.27](./1.27-pcap-assertions-ci.md)).

## Deliverables
1. Topology build/teardown in `tools/netns-nat-matrix.sh`.
2. Per-cell NAT-flavor application (all four cells).
3. Coturn + rendezvous running inside the topology with real ephemeral-credential minting.
4. A generic (non-Meridian) connectivity smoke check per cell proving each NAT flavor and TURN
   reachability behave as intended (e.g. a raw UDP/TCP probe or `turnutils_uclient`/STUN client) — this
   is independently testable without 1.24 landed.

## Status (done)
`tools/netns-nat-matrix.sh` was rewritten from the T05-era stub into a real six-namespace rig
(`ns-alice`/`ns-natA`/`ns-net`/`ns-turn`/`ns-natB`/`ns-bob`), reusing `tools/netns-two-lans.sh`'s
`link_pair`/`bridge_attach` helper shape (duplicated, not sourced — that script has no source-guard and
unconditionally runs its own dispatch). A new helper, `apps/rendezvous/examples/fetch_turn_credentials.rs`,
mints one real ephemeral TURN credential from a live rendezvous through the actual `TurnReq`/`TurnGrant`
wire flow (`SignalingClient::request_turn_credentials`) — independently verified the returned `credential`
is exactly `base64(HMAC-SHA1(secret, username))`, never a hand-rolled substitute.

**Rendezvous placement** (the open `TODO: confirm`) is resolved: `meridian-rendezvous` runs inside `ns-net`
itself, bound to the bridge's own address (`203.0.113.1:8443` — a Linux bridge device can hold its own IP
while still forwarding between member ports), rather than adding a seventh namespace. connectivity-debugger
confirmed this is legitimate and mirrors the reference small-org deployment (rendezvous + coturn
co-located, per `docs/architecture/system-design.md`'s "two containers plus a database"), not a shortcut
that breaks anything ADR-8 requires (that ADR's active-passive-behind-a-VIP concern is about rendezvous-
replica HA, unrelated to sharing an L2 segment with TURN).

**NAT flavors** on `ns-natA`/`ns-natB` (identical, parameterized `apply_nat_flavor`, flushed/reapplied per
cell rather than a full topology rebuild): `full-cone` (static 1:1 SNAT/DNAT, unrestricted FORWARD both
directions), `port-restricted` (MASQUERADE + FORWARD ACCEPT lan→pub, `ESTABLISHED,RELATED`-gated pub→lan
against a default FORWARD DROP policy), `symmetric` (`MASQUERADE --random-fully`), `udp-blocked`
(port-restricted plus a UDP FORWARD DROP). connectivity-debugger independently re-verified these live: a
reply from the same remote IP but a different port is genuinely dropped under `port-restricted` (true
port-restricted-cone, not merely address-restricted), and the `nat_flavor_probe` (two UDP listeners in
`ns-turn`, one bound client socket in `ns-alice` sending to both, comparing observed external ports) tests
the right dimension — endpoint-independent mapping — for what ICE/STUN cares about, with no flakiness risk
(concurrent conntrack entries can't collide on one external port by kernel design).

Real coturn (4.6.1) runs in `ns-turn` with a rig-generated config (fresh secret each run, `mktemp -d`
scratch dir — never touches or edits the checked-in `infra/coturn/turnserver.conf` template). Per-cell
smoke checks: the existing in-process `meridian doctor` check (kept as an additional signal), the
NAT-flavor probe above, and a TURN-reachability probe that mints a real credential (from within `ns-alice`,
genuinely behind the NAT under test) and drives `turnutils_uclient` against coturn asserting a real relay
allocation — using the literal minted `(username, credential)` pair via `-u`/`-w` (not a raw secret),
exactly what a real client presents to its ICE agent. `udp-blocked` additionally *positively* asserts the
UDP path is genuinely blocked (not just skipped) before asserting TCP reachability still works.

Two `turnutils_uclient` client-mode deviations came up during implementation and were resolved by
connectivity-debugger: (1) an initial report that `-u`/`-w` (precomputed-credential mode) reliably failed
against this coturn build did not reproduce under direct re-testing (most likely a stale coturn process
from an earlier dev iteration using a different secret, not an algorithm or tool bug) — the probe now uses
`-u`/`-w` with the real minted credential, which is strictly more faithful than the earlier `-W`
(raw-secret) workaround; (2) `-y` (client-to-client) mode empirically drives 4-5 concurrent allocations,
exceeding the real `user-quota=4` (1.14's tuned production value, kept as-is) — worked around with
single-allocation mode against a real `turnutils_peer` companion process, which is still a sound
reachability proof (real ALLOCATE/PERMISSION/relay-data path, matching the production client shape
`turnserver.conf`'s quota comment describes) and loses nothing this task needs.

test-engineer's review was cut short by a session limit, but before failing it found one real bug, which
was fixed and independently verified: `ensure_topology` restarts coturn and rendezvous independently and
idempotently (each skipped individually if already running), but the two share one HMAC secret; if coturn
alone died and was restarted, the original per-`start_coturn`-call `gen_secret()` minted a *fresh* secret
that no longer matched what the still-running rendezvous process already had baked into its config,
silently invalidating every credential from then on. Fixed with `get_or_create_secret()` — persists the
secret to the rig's scratch dir on first generation, and both `start_coturn`/`start_rendezvous` now read it
back rather than regenerating — verified by deliberately killing coturn's real PID, confirming port 3478
stopped listening, re-invoking `cell`, and confirming coturn restarted (new PID) with the *same* secret and
the reachability probe passed. The remaining scenarios from the review brief (`need_root()`'s graceful skip
as non-root, an unknown cell name, `down` called with nothing ever brought up) were spot-checked directly
and are already correct — no further fixes needed.

Independently verified end-to-end multiple times in this sandbox (root + `CAP_NET_ADMIN` available): all
four cells' smoke checks pass; `down` leaves zero stray namespaces, processes, or veths; a full `matrix` →
`down` → `matrix` → `down` cycle is reproducible; `cargo fmt --check` and
`cargo clippy --examples --all-targets -D warnings` clean on `meridian-rendezvous`.
