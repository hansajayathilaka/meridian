# 1.22 — `meridian` CLI: `--transport webrtc` wiring

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F11 wire, prerequisite) · **Depends on:** 1.15 · **Status:** done

## Goal
`meridian chat`/`session demo` can drive a real `WebRtcTransport` connection between two OS processes,
not only the deterministic `LoopbackTransport`. 1.15 proved the real backend works via a gated `cargo
test` inside `apps/core`; nothing in the CLI can select it yet. This is the direct prerequisite for
[1.23](./1.23-netns-nat-matrix.md)'s netns/tcpdump matrix, which needs two real cross-process peers to
capture packets from.

## Scope
- **In:** a `--transport <loopback|webrtc>` flag (or equivalent config knob) on the commands that build a
  `Transport` today (`meridian chat`, `meridian session demo`, `meridian doctor` where relevant); wire it
  to construct a `WebRtcTransport` behind the existing `webrtc` cargo feature instead of
  `LoopbackTransport`; a two-process (or two-task) integration test proving two real `meridian` instances
  connect, exchange chat, and report `transport=webrtc-datachannel` in `session info`, entirely on
  localhost (no NAT emulation, no netns, no coturn changes).
- **Out:** any NAT/namespace topology, coturn orchestration, or tcpdump capture (→ 1.23); any change to
  `WebRtcTransport` itself or the `Transport` trait (→ 1.15's successors if a gap is found — route through
  **architect** per that task's own note, don't guess a trait change here).

**Scope corrections (see Status):** the "In" line above, written before implementation, names three
commands (`chat`, `session demo`, `doctor`) as candidates for the flag. Investigation found only
`session demo` actually builds a `Transport` today:
- `meridian chat` relays E2EE chat envelopes through the rendezvous (T03) and never touches `Transport`
  at all — wiring P2P into `chat` is a distinct future feature, not this task's job, and stays untouched.
- `meridian doctor` is inherently a **simulated**-NAT-matrix diagnostic: it exists specifically to
  reproduce the four NAT cells (full-cone/port-restricted/symmetric/udp-blocked) deterministically via
  `LoopbackTransport`'s scenario simulation. `WebRtcTransport` is a real network backend with no
  NAT-scenario concept, so `--transport` doesn't meaningfully apply to it; `doctor.rs` is left untouched.

So the flag lands only on `meridian session demo`, which is where the real construction choice exists.

## Deliverables
1. `--transport` CLI flag/config knob selecting `webrtc` vs. `loopback`, gated behind the existing
   `webrtc` cargo feature (a non-feature build must still compile and default to loopback).
2. Integration test (or documented manual demo script) proving two real `meridian` processes connect
   over `WebRtcTransport` and chat, matching 1.15's headline "servers out of the data path" demo but
   driven through the CLI instead of a `cargo test` inside `apps/core`.

## Risks / notes
Keep this scoped to plumbing: the transport already exists and is tested (1.15); this task's job is
making the CLI able to choose it, not proving anything new about NAT/relay behavior (that's 1.23's job,
against real networks). If the CLI's command structure makes a clean two-process test awkward (e.g. the
binary isn't easily driven headlessly), a same-process two-task test using the CLI's own connection-setup
code path is an acceptable substitute — just don't quietly fall back to testing `apps/core` directly
again, which 1.15 already covered.

## Tests
`cargo nextest run -p meridian-cli --features webrtc` · the two-peer real-connection demo/test.

## Reviews
**rust-dev** implements; **test-engineer** (integration test); **connectivity-debugger** if any
connection-setup issue surfaces.

## Status (done)
Added a `--transport <loopback|webrtc>` flag to `meridian session demo` (`TransportArg` enum in
`apps/cli/src/main.rs`, following the existing `StoreArg` pattern) and a `webrtc` cargo feature on
`meridian-cli` (`webrtc = ["meridian-core/webrtc"]`, off by default).

`apps/cli/src/session.rs` was refactored so the ~40 lines of X3DH/bundle setup and dial/answer
plumbing are shared by both backends: a new private `establish<T: Transport>(ta, tb, policy,
ice_servers)` does everything the old `connect()` did except build the transports; the public
`connect(scenario, policy, ice_servers)` (used unchanged by `doctor.rs` and the loopback demo path)
now just builds a `LoopbackFabric`/`LoopbackTransport` pair and calls `establish`; a new
`#[cfg(feature = "webrtc")] connect_webrtc(policy, ice_servers)` builds a `WebRtcTransport::new()` pair
and calls the same `establish`. The printed-output logic (headline lines, chat exchange, ping, the
`session info`/`transport=...` line) was extracted into `run_demo_generic<T: Transport>`, shared
verbatim by both backends — so the `transport=webrtc-datachannel` vs. `transport=loopback` line in
`session info` is the same `SessionInfo::Display` impl either way, not a hand-rolled duplicate.

`run_demo` dispatches on `opts.transport` and fails closed: under `--transport webrtc`, a non-default
`--nat` or a non-`direct` `--policy` is rejected with a clear `Err` (not silently ignored) — there is no
NAT simulation for a real transport, and the demo's fabricated TURN servers
(`turn-a`, understood only by `LoopbackTransport`'s simulation) would make a real `WebRtcTransport`
actually try to reach a nonexistent host under `prefer-relay`/`relay-only`; real TURN reachability is
1.23's job. Without the `webrtc` feature compiled in, `--transport webrtc` fails closed with a clear
message naming the feature to rebuild with (matching `init_os_keystore`'s existing idiom in `main.rs`),
so a non-feature build still compiles and defaults to loopback.

New test file `apps/cli/tests/session_demo_webrtc.rs` (`#![cfg(feature = "webrtc")]`, matching
`apps/core/tests/p2p_session_webrtc.rs`'s gating), driving the real compiled `meridian` binary: asserts
the same headline claims as `session_demo.rs` (DTLS fp verified, rendezvous stopped, both chat lines)
plus `transport=webrtc-datachannel`, a JSON-mode check, and two validation-guard tests (`--nat
symmetric` and `--policy relay-only` under `--transport webrtc` both exit non-zero with an explanatory
stderr message).

Tests: `cargo build --workspace` clean; `cargo test -p meridian-cli` 14/14 pass (the new webrtc test
file compiles to 0 tests, correctly gated off); `cargo test -p meridian-cli --features webrtc` 18/18
pass (14 existing + 4 new); `cargo clippy -p meridian-cli --all-targets -- -D warnings` and `cargo
clippy -p meridian-cli --features webrtc --all-targets -- -D warnings` both clean; `cargo fmt --check`
clean. (`cargo-nextest` is not installed in this sandbox; `cargo test` was used as the equivalent
narrowest-loop runner — behaviorally identical for this crate's tests, which have no nextest-specific
config.)
