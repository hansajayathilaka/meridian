# 1.7 — SecretStore KDF op: drop the signature-determinism dependency

**Phase:** 1 — Review of Phase 0 · **Feature:** review fix (F7) · **Depends on:** — · **Status:** done

## Goal
The at-rest session-store key no longer silently depends on Ed25519 signature determinism, so a future
HSM/enclave `SecretStore` with randomized signatures can never make all persisted ratchet state
permanently undecryptable.

## Scope
- **In:** add a dedicated key-derivation operation to the `SecretStore` trait
  (`apps/store/src/lib.rs:70-83`) whose output does not rely on signature determinism; migrate
  `apps/crypto/src/at_rest.rs:19-24` off "sign a fixed label" onto the new op; resolve the existing
  `TODO: confirm`; document the derivation as a hard recoverability dependency until the backend contract
  is fully proven.
- **Out:** implementing an actual HSM/enclave backend; changing on-disk format for existing data beyond
  what the migration requires (call out any migration need explicitly).

## Deliverables
1. New derive-key method on the `SecretStore` trait + the loopback/default impl.
2. Edited `apps/crypto/src/at_rest.rs` using the new op.
3. Recoverability note in the store/crypto docs; resolved `TODO: confirm`.
4. Test that the derived store key is stable across calls and independent of signature randomness.

## Risks / notes
This is a trait-contract change — route to **architect**. If it changes the derived key for existing
stores, either preserve backward compatibility or document the migration; do not silently orphan data.
Keep the derivation domain-separated from message/root keys.

## Result
Added `SecretStore::derive_key(&self, h: &KeyHandle, info: &[u8]) -> Result<[u8; 32]>` — an
additive trait method alongside `store`/`use_key`/`nonextractable` — implemented in all three
software stores (`file.rs`, `mem.rs`, `os.rs`) via a shared `derive_key_from_seed(seed, info)` free
function in `apps/store/src/lib.rs` doing HKDF-SHA256 (fixed zero salt, raw seed as IKM, caller
`info` as the Expand info), mirroring `meridian-crypto`'s existing HKDF helper pattern and pinned
to the same workspace `hkdf` version. Zeroization of the raw seed is preserved identically to the
existing `use_key` path in each store.

`apps/crypto/src/at_rest.rs` now calls `store.derive_key(handle, STORE_KEY_INFO)` directly instead
of the old flow (sign a fixed label through the store, then HKDF the deterministic Ed25519
signature into a key) — `derive_store_key` and `STORE_KEY_LABEL` are removed, replaced by
`STORE_KEY_INFO`. This resolves the `TODO: confirm` and closes F7: the store key derivation no
longer depends on any signature algorithm's determinism, so a future HSM/enclave `SecretStore`
backend with randomized signatures can never make persisted ratchet state permanently
undecryptable. The one other call site (`apps/core/src/chat.rs::store_key`) was migrated to the
new call; a workspace-wide grep confirmed no other references to the old names remain.

**Breaking change (documented, not silently introduced):** this changes the derived session-store
key for any previously-persisted session-store blob — old blobs were sealed under
`HKDF(fixed-salt, Ed25519-signature-over-label, info)`, new blobs under `SecretStore::derive_key`'s
HKDF-Expand directly over the raw seed. No migration shim is provided. Confirmed acceptable
pre-release: no deployed users or persisted production data exists yet. Called out explicitly in
`at_rest.rs`'s module doc.

`docs/api/core-api-contracts.md`'s canonical `SecretStore` trait listing was updated to include
`derive_key` so it doesn't silently drift from the code.

**Tests added:** `apps/store/tests/keystore.rs::derive_key_is_stable_and_independent_of_signing`
(determinism across calls, domain separation by `info`, non-equality with signature bytes) and
`apps/crypto/tests/at_rest.rs` (two tests: the store key matches a hand-computed HKDF-Expand over
the raw seed independent of any signing call, and a seal/open round trip through the new
`derive_key` path).

**Verification:** `cargo test -p meridian-store -p meridian-crypto` all pass (including the new
tests), `cargo test --workspace` green, `cargo clippy --workspace --all-targets -- -D warnings`
clean, `cargo fmt --all -- --check` clean.

**Design review (architect, pre-implementation):** PASS — confirmed no ADR (0011/0015) governs
at-rest storage so no new/superseding ADR is needed; the dependency graph stays acyclic (`hkdf` is
a primitive crate, not a new workspace edge); an additive trait method is the right shape for a
stable platform-shim seam; `derive_key` returning raw HKDF-Expand output is a better fit for real
HSM/enclave backends than the old sign-then-HKDF scheme (most enclave APIs expose a raw-derive
primitive; guaranteeing deterministic signatures is something many HSMs explicitly refuse); the
breaking format change is acceptable pre-release with no compatibility obligation. Flagged three
required doc-sync edits, all landed (see above).

**Reviews (post-implementation):** **architect** PASS (confirmed the trait change is purely
additive with no other `SecretStore` implementers affected, all three required doc-sync edits
landed, dependency graph still acyclic, ADR scoping held — diff didn't drift into X3DH/ratchet
territory). **security-reviewer** PASS (confirmed domain separation is sound — the seed is never
used as HKDF IKM anywhere else in the crate, X3DH/ratchet HKDF calls always take DH outputs as IKM,
never the raw seed, so there's no cross-construction collision surface; recoverability is
genuinely improved since the new construction has no RNG or signature dependency; the
breaking-change handling is honest and appropriately scoped to pre-release; key hygiene/zeroization
preserved in all three store impls; no oracle-leak or logging concerns).
